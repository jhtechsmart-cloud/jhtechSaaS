# 고객 등록 — 필수항목 강화 + 중복 방지 설계

- 작성일: 2026-07-13
- 상태: 설계 확정(구현 계획 대기)
- 관련 코드: `apps/web/src/app/admin/customers/**`, `apps/web/src/lib/customers/**`, `packages/shared/src/biz-no.ts`, `supabase/migrations/**companies**`

## 한 문장 요약

고객 등록에서 **사업자번호를 기본 필수·유일키로 삼고**(예외는 '사업자번호 없음' 명시), **대표자·연락처·주소도 필수**로 올리며, **입력하는 순간 기존 업체를 실시간으로 찾아 경고하고 저장 시 서버가 최종 차단**해서 중복·누락을 막는다.

## 왜 필요한가 (비전문가용)

고객 데이터는 견적·AS·소모품 신청이 전부 매달리는 뿌리다. 여기에 같은 업체가 이름만 조금 다르게 두 번 등록되면(중복), 견적 이력·미수금·AS 기록이 두 곳으로 갈라져 데이터가 망가진다. 지금 실제로 "다른 회사이름 + 같은 사업자번호"가 중복 등록되는 일이 발생했다.

## 현재 문제 (원인 규명)

코드/마이그레이션 조사 결과:

- 사업자번호(`biz_no`)는 저장 전 하이픈이 제거되어 정규화되고, DB에 부분 UNIQUE 인덱스도 있다: `companies_biz_no_unique on companies (biz_no) where biz_no is not null` (`supabase/migrations/20260602100001_companies.sql:23`).
- **그러나 사업자번호는 선택 항목이다** (`apps/web/src/lib/customers/schema.ts` — `name`만 필수). 값을 비우면 `null`로 저장된다 (`apps/web/src/lib/customers/actions.ts` `biz_no: v.biz_no ? normalizeBizNo(...) : null`).
- 부분 UNIQUE는 `where biz_no is not null`이라 **`null`인 행끼리는 유일성 검사 대상에서 완전히 제외** → 회사명만 조금 다르게 해서 같은 업체를 몇 번이든 중복 등록 가능.
- 등록 시 "이미 있는 업체인가?" **사전 조회가 없고**, 오직 DB unique violation(23505)을 잡아서만 반응한다. 사업자번호가 비어 있으면 그 방어도 안 걸린다.
- 회사명·전화 등 다른 어떤 필드에도 중복 방지 장치가 없다.

**결론: "UNIQUE가 뚫린" 게 아니라, 사업자번호를 안 넣으면 애초에 방어 대상이 아니었던 것.**

> 별도 확인 필요(구현 1단계): 만약 실제 중복 두 행에 **동일한 10자리 사업자번호가 둘 다** 들어가 있다면, prod에 UNIQUE 인덱스 마이그레이션이 적용되지 않은 것이므로 인덱스 추가가 필요하다. prod에서 `select indexdef from pg_indexes where indexname='companies_biz_no_unique'` 및 중복 두 행의 `biz_no` 실제 값을 확인한다.

## 목표 / 비목표

**목표**
- 관리자 고객 등록/수정 시 필수항목 강화(누락 방지).
- 사업자번호 기반 중복을 실시간+저장 시점 이중으로 차단.
- 사업자번호 없는 예외 고객도 회사명+연락처로 중복 경고.
- 기존 중복 의심 목록을 화면에서 확인(수동 정리용).

**비목표(이번 범위 밖)**
- 중복 업체 **병합(merge)** 기능(FK 재연결·이력 이관) — 별도 스펙.
- ceo·address·phone의 DB `NOT NULL` 강제 — 기존 빈 데이터·예외 때문에 앱 레이어로 대신함.
- 공개 신청(anon)·견적요청 가져오기·수기견적 경로의 재설계(이미 사업자번호 dedup 존재).

## 접근 방식: A — 앱 레이어 중심 (확정)

필수·중복 검증을 **Zod + 서버액션 + 실시간 조회**로 강제한다. DB는 사업자번호 부분 UNIQUE(이미 있음)를 유지·확인만 하고, 다른 컬럼에 `NOT NULL`을 추가하지 않는다.

- 이유: 기존 데이터에 빈 값·중복이 이미 있어 DB `NOT NULL`을 바로 걸면 마이그레이션이 실패하거나 기존 데이터를 급히 손봐야 한다. '사업자번호 없음' 예외와 공개접수 흐름(company_id null)과도 충돌한다.
- 사업자번호가 있는 정상 케이스는 앱(필수·실시간·저장검증) + DB(부분 UNIQUE)로 이중 보호된다.

## 상세 설계

### 1) 필수항목 (앱 레이어: 화면 `*` + Zod + 서버 재검증)

필수 = **업체명 · 사업자번호 · 대표자 · 연락처(최소 1개) · 주소**

- 업체명(`name`): 기존 유지.
- 사업자번호(`biz_no`): 필수 + **10자리 + 체크섬 통과**. 단 '사업자번호 없음' 체크 시 예외(아래 2).
- 대표자(`ceo`): 필수(trim 후 1자 이상).
- 연락처: **휴대폰(`mobile`)·전화1(`phone1`)·대표연락처(`phone`) 중 최소 1개**가 채워져야 함. 순수함수 `hasAnyContact()`로 판정.
- 주소(`address`, 사업장 주소): 필수(trim 후 1자 이상).
- 이메일(`email`): 필수 아님. 단 **값이 있으면 형식 검증**(`z.string().email()` 또는 동등) 추가.

화면: 위 필수 필드 라벨에 빨간 `*` 표시. Zod 스키마(`companyFormSchema`)에서 `.min(1)`/커스텀 refine으로 강제. 서버액션(`createCustomer`/`updateCustomer`)이 동일 스키마로 재검증.

### 2) 사업자번호 '없음' 예외

- 폼에 **`사업자번호 없음` 체크박스** 추가.
- 체크 시: 사업자번호 입력칸 비활성+선택 처리. 스키마는 biz_no 필수 조건을 면제하고 `null` 저장 허용.
- 미체크(기본): biz_no 필수 + 형식 검증.
- 저장 payload에 예외 여부가 드러나도록 처리(예: biz_no가 빈 값이면서 '없음' 플래그가 true여야 통과. 플래그 없이 빈 값이면 검증 실패).

### 3) 중복 확인 프로세스 (실시간 경고 + 저장 차단)

**실시간(클라이언트)**
- 사업자번호 10자리가 완성되면 기존 업체 조회. 기존 RPC `lookup_company_by_biz_no`(`supabase/migrations/20260602100003_lookup_company_by_biz_no.sql`) 활용/확장.
- 발견 시 경고 카드: *"이미 등록된 업체: {회사명} (대표 {대표자}) · 담당 {담당영업}"* + `[기존 업체 열기]` 링크. **저장 버튼 잠금.**
- '없음' 고객: 회사명+연락처가 채워지면 유사 업체 조회(회사명 정규화 일치 + 전화 일치) → 동일하면 경고.
- 디바운스로 과호출 방지. 수정 폼에서는 **자기 자신(같은 id)은 중복에서 제외.**

**저장 시(서버, 최종 방어선)**
- 서버액션이 저장 직전 다시 조회(실시간 조회와 저장 사이 경합 방지) + DB unique violation을 잡아 이중 차단.
- 중복이면 저장 거부 + 기존 업체 id/이름을 응답에 담아 화면이 "기존 업체로 이동" 안내.

### 4) 적용 범위

- 고객 **등록** 폼 + `createCustomer`.
- 고객 **수정** 폼 + `updateCustomer`(사업자번호를 바꿀 때도 중복 검사, 자기 자신 제외).
- 공용 컴포넌트 `CompanyForm`(create/edit 공용)에서 필수·예외·실시간 경고를 함께 처리.

### 5) DB / 데이터 관점

- **prod 인덱스 확인(구현 1단계)**: `companies_biz_no_unique` 존재 여부 + 중복 두 행 biz_no 값 확인. 없으면 인덱스 추가 마이그레이션 작성.
- 정규화(하이픈 제거, `normalizeBizNo`)는 이미 있음 — 유지.
- ceo·address·phone은 DB `NOT NULL` 추가하지 않음(앱 레이어 필수).

### 6) 기존 중복 처리 = 목록 리포트만

- 병합 없음. 다음을 뽑는 **중복 의심 목록**(읽기 전용 화면 또는 스크립트):
  - 동일 사업자번호 2행 이상(있다면 인덱스 미적용 신호),
  - 사업자번호 빈 값 고객,
  - 회사명 정규화 일치(공백·괄호 제거) 다중 행.
- 운영자가 화면에서 하나씩 보고 수동 정리.

### 7) 곁다리 개선

- 이메일 형식 검증(값이 있을 때만) — 사용자 승인으로 이번에 포함.

## 컴포넌트/모듈 경계

- `packages/shared` 또는 `apps/web/src/lib/customers/`에 순수 판정 함수 신설:
  - `hasAnyContact(fields)` — 연락처 최소 1개 판정.
  - `normalizeCompanyName(name)` — 회사명 정규화(중복 비교용).
  - 필수/예외 규칙을 반영한 `companyFormSchema` 리팩터(refine).
- 실시간 중복 조회 = 기존 `lookup_company_by_biz_no` 재사용(+ 회사명/전화 조회가 필요하면 확장 또는 신규 조회 액션).
- `CompanyForm`은 검증·경고 상태를 표시만 하고, 판정 로직은 위 순수 함수/서버로 위임.

## 검증 계획

- **단위(Vitest)**: `hasAnyContact`, `normalizeCompanyName`, biz_no 필수/예외 refine, 이메일 형식 검증.
- **e2e(Playwright)**: 등록 시 필수 누락 → 저장 불가 / 기존 사업자번호 입력 → 실시간 경고+저장 차단 / '없음' 예외 등록 / 수정 시 자기 자신 제외.
- **db-tests(RLS/제약)**: 동일 biz_no 두 번 INSERT → unique violation.
- 게이트: web typecheck·lint0·unit·build·e2e.

## 열린 항목(구현 중 확정)

- prod `companies_biz_no_unique` 실제 존재 여부(→ 인덱스 추가 마이그 필요 여부 결정).
- '없음' 예외 플래그를 payload에 어떻게 표현할지(임시 폼 필드 vs biz_no=null+명시 플래그) — 구현 시 최소 변경으로.
- 회사명+연락처 중복 경고의 민감도(정확 일치만 vs 유사) — 우선 정확 일치로 시작.
