# 관리자 페이지 모바일 대응 — 설계 (Spec)

**작성일**: 2026-06-17 · **상태**: 승인됨 → 구현 대기

## 한 문장 요약
데스크톱 화면(`lg`=1024px 이상)은 지금 그대로 두고, 그 미만(모바일·태블릿 세로)에서만 레이아웃이 "모바일 모드"로 바뀌도록 — 사이드바는 햄버거 드로어로, 의뢰관리 2분할은 목록→상세 전환으로, 견적 합계는 하단 고정 바로 — 단계별로 손본다.

## 왜 필요한가
지금 코드는 Tailwind 반응형이라고 해도, **고정 너비 사이드바(224px)**, **의뢰관리 2분할(목록 300px + 상세)** 같은 셸 구조가 폰 폭(375~390px)에 안 들어가서 모바일에서 사실상 못 쓴다. 그런데 영업 담당자는 외근·미팅 현장에서 **견적 작성, 수기 견적, 견적 메일 발송, 고객 정보 확인·응대**를 모바일로 더 자주 하게 된다(미팅 메모 참조: "모바일 즉시 발송"이 핵심 요구). 데스크톱을 건드리지 않으면서 모바일에서도 이 흐름을 쓸 수 있게 만드는 것이 목표.

## 핵심 원칙
- **데스크톱 무손상**: 모든 분기는 Tailwind `lg:`(1024px) 기준. `lg` 이상은 기존 코드 경로 그대로 → 회귀 위험 최소.
  - 태블릿 세로(768~1023px)도 모바일 모드로 들어간다(좁은 폭에서 2분할이 무리이므로 의도된 동작).
- **전환용 UI 상태는 쿠키 안 씀**: 드로어 열림·목록/상세 전환은 매 로드마다 기본 상태(닫힘/현재 라우트)에서 시작하므로 서버·클라 초기값이 일치 → **hydration mismatch(서버·클라 렌더 불일치) 위험 없음**. (영속이 필요한 UI 상태만 쿠키 — CLAUDE.md 규칙.)
- **최소 변경**: 기존 컴포넌트를 갈아엎지 않고, 모바일 분기 클래스·래퍼·드로어만 더한다.

## 범위
포함(우선순위 순):
1. 공통 셸(사이드바·상단바) — 나머지가 다 그 위에서 동작하는 토대
2. 의뢰관리 2분할 (견적 작성·고객 응대의 무대)
3. 견적 작성·수기 견적 (하단 고정 합계 바)
4. 대시보드 (캘린더·차트 가로 넘침)
5. 메일 모달 + 고객 상세 (이미 거의 대응 → 폴리시)

범위 밖:
- **출고의뢰서**: 아직 작성 폼 UI가 없고 별도 세션에서 개발 중. 여기서 구현하지 않고, 그 설계 문서(`2026-06-17-release-order-design.md` §4)에 "폼을 처음부터 반응형으로" 조항을 추가했다. 실제 구현은 그 세션 담당.
- 관리자 기타 설정 화면(카테고리·사용자·장비 등) — 모바일 사용 빈도 낮음. 추후 필요 시 별도.

## Phase 1 — 공통 셸: 햄버거 드로어

**대상 파일**: `apps/web/src/app/admin/layout.tsx`, `apps/web/src/app/admin/_components/AdminSidebar.tsx`

**현재**: `<aside>`가 `w-56`(펼침 224px)/`w-16`(접힘 64px) 고정 너비로 항상 본문 옆에 붙어 있다. `md:hidden` 등 모바일 분기 없음.

**모바일(`lg` 미만) 동작**:
- 고정 사이드바를 숨긴다(`hidden lg:flex`).
- 상단바 좌측에 `☰`(햄버거) 버튼을 추가(`lg:hidden`).
- `☰` 누르면 사이드바가 왼쪽에서 슬라이드로 들어오는 **오버레이 드로어**(fixed, z-index 상단, 뒷배경 backdrop 어둡게).
- 닫힘: 배경(backdrop) 탭, 메뉴 항목 선택, `Esc`.
- **항상 닫힌 채로 시작** → 쿠키 불필요.
- 드로어 안의 메뉴 항목은 **기존 AdminSidebar 메뉴를 그대로 재사용**(메뉴 추가돼도 자동 반영).

**데스크톱(`lg` 이상)**: 지금 그대로 — 고정 사이드바 + 접기 토글 + hover 펼침 + 쿠키(`jh.sidebarCollapsed`).

**분리/구조**: 드로어 열림 상태(boolean)는 layout의 클라이언트 셸에서 관리. AdminSidebar는 "고정 모드 / 드로어 모드" 양쪽에서 같은 메뉴를 렌더할 수 있게 props로 분기(내부 메뉴 데이터는 단일 출처 유지).

## Phase 2 — 의뢰관리: 목록→상세 전환 (master-detail)

**대상 파일**: `apps/web/src/app/admin/applications/layout.tsx`, `apps/web/src/app/admin/applications/_components/ApplicationListPane.tsx`, 의뢰 상세 페이지

**현재**: `ApplicationsLayout`이 `flex h-[calc(100dvh-57px)]`로 좌측 목록 패널(`w-[300px]` 고정) + 우측 상세(children)를 **항상 동시에** 렌더. 폰 폭에선 목록 300px만으로 거의 다 차서 상세가 75px만 남음.

**모바일(`lg` 미만) 동작** — 목록→상세 전환:
- **목록 라우트**(`/admin/applications`, 선택된 상세 없음): 의뢰목록 패널만 풀화면.
- **상세 라우트**(`/admin/applications/[id]`): 상세만 풀화면 + 상단에 `‹ 목록` 뒤로가기(→ `/admin/applications`).
- **판단 방법**: layout의 클라이언트 래퍼가 `usePathname()`으로 현재 경로가 상세(`/admin/applications/<id>`)인지 목록인지 판별 → 목록 패널과 상세 영역에 표시/숨김 클래스를 토글.
  - 목록 패널: 상세 열림 시 `hidden`, 목록일 때 `flex w-full`. `lg`에선 항상 `lg:flex lg:w-[300px]`.
  - 상세 영역: 목록일 때(모바일) `hidden`, 상세일 때 `block w-full`. `lg`에선 항상 표시.
- 고정 높이 `h-[calc(100dvh-57px)]`: 모바일에선 단일 패널이 그 높이 안에서 자체 스크롤되게 유지(가로 넘침 방지).

**데스크톱(`lg` 이상)**: 지금 2분할 그대로(목록 300px + 상세).

**주의**: 의뢰 상세 본문은 이미 `grid-cols-1` → `lg:grid-cols-[1fr_320px]`로 단열 전환됨(요약 패널). 그 위에 아래 Phase 3의 하단 바를 얹는다.

## Phase 3 — 견적 작성·수기 견적: 하단 고정 합계 바

**대상 파일**: 의뢰 상세 견적 프레임(`.../[id]/_components/quote-frame/*`, `QuoteSummaryPanel`), 수기 견적 폼(`apps/web/src/app/admin/quotes/_components/ManualQuoteForm.tsx`, `QuoteTotalsAside`), 줄 편집기(`QuoteLinesEditor`).

**현재**: 폼은 `grid-cols-1` → `lg:grid-cols-[1fr_320px]`로, `lg` 미만에선 우측 합계 요약(sticky 320px)이 **맨 아래로 밀린다** → 장비·옵션 입력 중 합계가 안 보임.

**모바일(`lg` 미만) 동작**:
- 화면 하단에 **고정 바**(`fixed bottom-0`, `lg:hidden`): `공급가 ₩×××  [발행/저장]`.
  - 표시 금액은 **공급가(`supply_price`, VAT 별도)** — 화면 표시 통일 규칙(CLAUDE.md E8) 준수. 부가세 미표시, "VAT 별도" 안내 문구 유지.
  - 액션 버튼은 해당 화면의 주 액션(견적 발행 / 수기견적 저장)을 그대로 연결.
- 흐름 안의 상세 요약 카드(품목별 내역)는 그대로 두되, **상시 합계 + 주 액션만 하단 바로 승격**(중복 액션은 피한다 — 흐름 안 버튼과 하단 바 버튼이 같은 핸들러).
- **줄 편집기(QuoteLinesEditor) 가로 넘침 점검**: 장비·옵션·금액이 한 줄에 들어가는 테이블이면 좁은 폭에서 넘친다 → 가로 스크롤 래퍼 또는 모바일 카드형 중 구현 시 확인(기본은 가로 스크롤). *(구현 디테일 — 실제 컴포넌트 보고 결정)*

**데스크톱(`lg` 이상)**: 우측 sticky 320px 요약 그대로, 하단 바 없음.

## Phase 4 — 대시보드: 캘린더·차트 가로 넘침

**대상 파일**: `apps/web/src/app/admin/dashboard/_components/TwoWeekCalendar.tsx`, `WeeklyUnitChart.tsx`

**현재**: KPI 카드(`sm:grid-cols-2 xl:grid-cols-4`)와 우측 레일(`xl:grid-cols-[1fr_330px]`, lg 이하 아래로 reflow)은 이미 대응됨. **2주 캘린더와 주간 활동 차트가 `grid-cols-7` 고정** → 폰에서 칸이 ~40px 이하로 뭉개진다.

**모바일(`lg` 미만) 동작**:
- 기본안: 7열 그리드를 **가로 스크롤 래퍼**(`overflow-x-auto`)로 감싸고 셀에 `min-w`를 줘서 칸이 읽을 만한 크기 유지. 데스크톱은 영향 없음.
- **검토 대안(구현 전 확정)**: 모바일에선 캘린더 대신 **"다가오는 일정 리스트"(agenda 뷰)** 로 바꾸는 게 더 나을 수 있음 — 가로 스크롤보다 손가락 친화적. 구현 비용이 더 크므로, 기본은 가로 스크롤로 적어두고 실제 구현 시 캘린더 데이터 구조 보고 최종 결정.

**데스크톱(`lg` 이상)**: 지금 그대로.

## Phase 5 — 메일 모달 + 고객 상세 폴리시

**대상 파일**: `.../quote-frame/SendQuoteEmailModal.tsx`, `apps/web/src/app/admin/customers/[id]/*`(`CustomerInfoSidebar.tsx`, `CustomerHeader.tsx`)

**현재**: 메일 모달은 이미 `w-full max-w-lg p-4`로 거의 대응됨. 고객 상세도 `grid-cols-1` → `lg:grid-cols-[330px_1fr]`로 모바일에선 단열로 쌓임.

**모바일 동작(폴리시 위주)**:
- 메일 모달: 실제 폰에서 입력·키보드 동작 점검. textarea가 키보드에 가리지 않게(높이/스크롤) 미세 조정.
- 고객 상세 `CustomerInfoSidebar`의 `FieldRow`: 라벨(`w-24`)+우정렬 값 구조에서 긴 값이 잘림 → 좁은 폭에서 줄바꿈/말줄임 정리.

**데스크톱(`lg` 이상)**: 지금 그대로.

## 테스트 / 게이트
- **Playwright E2E (모바일 뷰포트 390px 추가)**:
  1. 드로어 — `☰` 열기 → 메뉴로 다른 화면 이동 → 닫힘.
  2. 의뢰관리 — 목록에서 항목 탭 → 상세 전환 → `‹ 목록` 뒤로가기.
  3. 견적 작성 — 하단 고정 바에 공급가 합계 보임 → 발행 동작.
  4. 대시보드 — 캘린더/차트 영역이 가로로 넘쳐도 가로 스크롤로 접근 가능(뷰포트 밖으로 레이아웃 깨짐 없음).
- **데스크톱 E2E**: 기존 그대로 유지(회귀 방지) — `lg` 이상 경로가 안 바뀌는지 확인.
- 순수 로직(드로어 토글 상태·`usePathname` 기반 master-detail 판단 등)은 가능하면 순수 함수로 빼서 Vitest 단위.
- **공통 게이트**(머지 전 전부 통과): `shared test`·`web test`·`db-tests test:rls`·`web typecheck`·`lint`·`build`·`web test:e2e`·`as any` 0.
  - 이번 작업은 대부분 DB 무변경(레이아웃) → 해당 단계는 `db push` 불필요. (단, db-test/e2e는 클린 `db reset` + `seed-local` 후 실행 규칙 준수.)

## 진행 방식 (각 Phase = 별도 PR)
1. 공통 셸(드로어)
2. 의뢰관리 master-detail
3. 견적 하단 고정 바
4. 대시보드 캘린더·차트
5. 메일 모달 + 고객 상세 폴리시

- 동시 세션이 흔하므로 각 Phase는 **worktree 격리**, 머지 전 `origin/main` 선병합으로 충돌 선해소(메모리 규칙). dev 시작 시 로컬 main이 뒤처져 있으면 `git pull --ff-only`.
- Phase는 독립적으로 머지 가능(서로 강결합 아님). 단, Phase 1(셸)이 모바일에서 화면 도달의 전제이므로 **1을 먼저** 머지하는 걸 권장.

## 미해결 / 구현 시 확정할 디테일
- Phase 4 캘린더: 가로 스크롤 vs agenda 리스트 (기본=가로 스크롤).
- Phase 3 줄 편집기: 가로 스크롤 vs 모바일 카드형 (실제 컴포넌트 보고 결정).
- 분기점 `lg`(1024px) 확정 — 태블릿 세로도 모바일 모드. (`md`=768px로 낮추면 태블릿은 데스크톱처럼 2분할 → 좁아서 비권장.)
